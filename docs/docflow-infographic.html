해결하는 문제 — Before vs After
Before — DocFlow 없이
개발 중 발견: 요구사항의 충돌 · 누락을 코딩 도중에야 알게 된다
문서 부재: 도메인 규칙, 정책이 코드에만 존재하고 문서화되지 않는다
흩어진 지식: 문서를 작성해도 체계 없이 흩어져 재활용이 어렵다
반복 작업: 매번 비슷한 구조의 문서를 처음부터 수동으로 작성한다
인수인계 고통: 개발 컨텍스트가 사라져 후임이 처음부터 파악해야 한다
기획서 수령 → 머리로 분석 → 놓친 부분 발생 → 개발 중 발견 → 재작업
→
After — DocFlow 사용 시
사전 탐지: AI가 충돌 · 엣지케이스 · 누락 정책을 개발 전에 발견한다
자동 생성: 요구사항만 쓰면 도메인 규칙, 시나리오, API 명세가 자동으로 나온다
체계적 축적: Vault에 프로젝트별로 구조화되어 언제든 조회 · 추적 가능하다
반복 제거: 파생 문서 작성에 들이던 반복 노동을 AI가 대신 수행한다
컨텍스트 보존: 축적된 문서가 "왜 이렇게 만들었는가"의 근거가 된다
요구사항 작성 → AI 분석 → 문서 보강 → 산출물 자동 생성 → Vault 축적

핵심 가치
🛡
누락 방지
AI가 요구사항의 충돌, 엣지케이스, 누락된 정책을 개발 전에 사전 탐지한다. 비용이 가장 적은 시점에 문제를 발견.
⚙
문서화 자동화
요구사항만 작성하면 도메인 규칙, 유스케이스, API 명세 등 6종의 파생 문서가 자동 생성된다.
📚
지식 축적
Vault에 프로젝트별로 구조화되어 쌓이는 문서가 살아있는 스펙이 된다. 인수인계, 회고의 근거.
🔄
반복 제거
매번 비슷한 구조의 문서를 수동으로 작성하는 반복 작업을 제거하여 본질적 업무에 집중.

왜 DocFlow인가 — 핵심 차별점
🧠
개발 전 리스크를 제거한다
코딩 후가 아니라 요구사항 단계에서 충돌 · 누락 · 엣지케이스를 발견한다. 수정 비용이 가장 낮은 시점에 문제를 해결하는 것이 DocFlow의 핵심 전략.
⚡
문서 작성 시간을 0에 가깝게
요구사항만 쓰면 6종의 산출물이 자동으로 나온다. 개발자는 생성된 문서를 검토하고 편집하는 것만으로 표준 문서 체계를 갖출 수 있다.
📚
지식이 사라지지 않는다
Obsidian Vault에 구조화되어 축적되는 문서는 프로젝트의 집단 기억이 된다. 인수인계, 기능 추가, 회고 시 "왜 이렇게 만들었는가"의 답이 항상 존재한다.
🔄
유연한 반복 워크플로
3단계 모두 선택적이고 반복 가능하다. 분석 없이 생성하거나, 생성 후 다시 작성으로 돌아갈 수 있다. 경직된 프로세스가 아니라 사용자의 작업 흐름에 맞춘다.
한 줄 요약: DocFlow는 "요구사항을 쓰면, 개발 문서가 나온다"는 단 하나의 약속을 지키기 위한 도구다. AI가 요구사항의 빈틈을 찾아주고, 파생 문서를 자동으로 만들어주고, 만든 문서를 체계적으로 쌓아준다. 개발자는 본질적 사고 — 무엇을 만들 것인가 — 에만 집중하면 된다.
