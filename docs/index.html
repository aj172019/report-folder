데이터베이스 설계

프로젝트 관리(PM)와 데이터베이스 설계(DB) 두 영역에 걸쳐 AI Agentic 프로세스를 도입한 결과, 전체 워크플로의 약 78%가 AI에 의해 자동화 또는 반자동화되었습니다. 중요한 점은 나머지 22%가 AI로 대체할 수 없는 고부가가치 영역이라는 것입니다. 문제 정의, 비즈니스 의사결정, 설계 방향 판단, 최종 품질 검증 등 "무엇을 해야 하는가"를 결정하는 영역은 여전히 사람의 몫이며, AI는 "어떻게 실행할 것인가"를 처리합니다.

영역 자동화 대상 AI 역할
PM - 데이터 관리 WBS JSON 수정, 상태 갱신, 공수 산정 자연어 명령 해석 + 규칙 기반 처리
PM - 시각화 대시보드, 간트 차트, KPI 카드 데이터 변경 감지 + 실시간 렌더링
PM - 보고 Google Sheets 동기화, 월별 시트 생성 자동 업로드 + 서식 적용
DB - 설계 스키마 구조 제안, 패턴 적용 설계 토론 파트너 + 일관성 검증

도구 비용과 복잡도의 불균형

Jira, Confluence 등 엔터프라이즈 도구는 인당 월 $7.75 이상의 라이선스 비용이 발생합니다. 10명 기준 연간 약 $930 이상이며, 이에 더해 관리자의 설정 및 유지보수 공수가 지속적으로 요구됩니다.

⏱

수동 현행화의 반복

Excel 기반 WBS는 매일 수동으로 상태를 갱신해야 합니다. 담당자별 진척 현황 수집, 공수 입력, 일정 변경 반영에 회당 30분 이상이 소요되며, 데이터는 갱신 시점 기준으로 항상 지연됩니다.

📄

데이터 산재와 보고 부담

WBS는 Excel, 진행 현황은 Jira, 커뮤니케이션은 메일/메신저에 분산됩니다. 주간/월간 보고를 위해 데이터를 수집/가공/시각화하는 별도 수작업이 반복적으로 발생합니다.

DB 영역

🔄

다이어그램-DB 불일치

변경이 반복될수록 다이어그램과 실제 DB의 동기화가 깨집니다. GUI 도구에서 다이어그램을 작성하고, DDL 스크립트를 수동 추출한 뒤, 데이터베이스에 수동 적용하는 과정에서 불일치가 누적됩니다.

🔍

설계 의도 소실

"왜 이렇게 설계했는가"를 그림만으로는 알 수 없습니다. 다이어그램에는 구조만 남고 설계 근거가 소실되어, 후임자나 타 팀원이 설계 의도를 파악하기 어렵습니다.

💰

높은 비용 + 협업 제약

전문 GUI 도구의 연간 라이선스 비용이 100~300만원에 달하며, 파일 기반 공유로 동시 편집이 불가능합니다. 설계 1건에 3~5일이 소요되는 생산성 문제도 존재합니다.

JOURNEY

문제 해결을 위한 탐색 과정

하나의 도구에 정착하지 않고, 각 단계에서의 한계를 인식하고 다음 단계로 발전시킨 과정입니다.

전환 완료

Jira + Confluence

프로젝트 초기 체계적인 이슈 트래킹을 위해 Jira를 도입했습니다. 스프린트 보드, 백로그 관리, 보고 기능을 활용했으나, WBS 수준의 세밀한 태스크 관리에는 커스터마이징 한계가 있었고, 전반적인 속도도 무거워 실제 활용도가 낮았습니다.

1
전환 완료

DA# + Redgate

GUI 도구에서 다이어그램을 작성하고, DDL 스크립트를 수동 추출한 뒤, 데이터베이스에 수동 적용하는 전통적 방식입니다. 다이어그램과 실제 DB의 불일치가 누적되고, 설계 의도가 소실되며, 높은 라이선스 비용과 협업 제약이 존재했습니다.

한계 확인

Excel + Google Sheets

Jira의 과도한 커스터마이징 한계와 무거운 속도 등의 이슈로 Excel 기반으로 전환했습니다. 속도와 유연성은 확보되었으나, 데이터 현행화가 전적으로 수작업에 의존하는 구조였습니다. 상태 변경, 일정 조정, 보고서 생성 모두 수동이며, 버전 관리가 불가능했습니다.

2
한계 확인

ChatGPT + dbdiagram.io

AI가 요구사항을 분석하여 DDL 초안을 자동 생성하고, dbdiagram.io의 간결한 DSL로 빠른 시각화가 가능해졌습니다. 그러나 수동 복사/붙여넣기, SaaS 종속, 설계와 코드의 단절, 맥락 단절 등의 한계가 남아 있었습니다.

현재 운영 중

AI Agentic WBS

자연어 명령으로 데이터를 관리하고, 시각화와 보고가 자동으로 이루어지는 구조를 설계했습니다. AI Agent에게 명확한 역할과 규칙을 부여하면, 프로젝트 관리의 반복 업무를 구조적으로 자동화할 수 있습니다.

3
현재 운영 중

AI Agentic JSON Spec

단일 원본(JSON 스펙)이 유일한 원본이 되고, 다이어그램/DDL/마이그레이션이 모두 여기서 자동 생성됩니다. AI를 설계 파트너로 활용하고, 설계 의도가 스펙에 보존되며, End-to-End 자동화 파이프라인을 구축했습니다.

설계 스펙의 차별점

기존 DDL은 데이터베이스에 "어떤 구조를 만들 것인가"만 기술합니다. 반면 설계 스펙(JSON)은 "왜 이렇게 설계했는가"까지 포함하는 의미 있는 설계 문서입니다.

기존: DDL (기술적 명세만)
CREATE TABLE "tb_product" ( "product_key" bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY, "product_name" varchar(200) NOT NULL, "product_type_ckey" integer NOT NULL, "sale_status_ckey" integer NOT NULL, "use_yn" char(1) DEFAULT 'Y' ); -- 이것만으로는 알 수 없는 것: -- product_type_ckey에 어떤 값이 들어가는지? -- 왜 use_yn이 필요한지? -- 이 테이블이 어떤 그룹에 속하는지?
AI Agentic: JSON Spec (의도 포함)
"tb_product": { "comment": "상품 마스터", "entityType": "master", "designNotes": ["매장-상품 매핑으로 다대다 관리"], "columns": { "product_type_ckey": { "type": "integer", "codeIds": { "카탈로그": { "상품": { "상품유형": { "단품": {}, "세트": {}, "옵션": {} } }}} } } } // 설계 의도, 코드 값 정의, 그룹 분류가 모두 포함

INSIGHTS
"AI 시대에 도구를 만드는 진입장벽은 빠르게 낮아지고 있습니다. 그러나 어떤 문제를 풀어야 하는지 정의하고, 해결 경로를 설계하며, 이를 반복 가능한 프로세스로 안착시키는 역량의 가치는 오히려 높아지고 있습니다."

도구가 아니라 프로세스

Phase 2에서 비용은 줄었지만 근본 문제는 남았습니다. 바뀌어야 할 건 도구가 아니라 일하는 방식이었습니다.

AI에게 규칙을, 사람에게 판단을

명확한 역할 분리와 단일 원본(JSON) 구조가 있어야 AI가 실행기가 아닌 파트너로 작동합니다.

직무 단위가 아닌 문제 단위로

작업을 직무 기준으로 나누면 사람에 종속됩니다. 문제 단위로 쪼개면 AI든 사람이든 가장 적합한 쪽이 처리할 수 있습니다.

외부 의존 제거

Jira, dbdiagram.io 등 외부 서비스를 걷어내고 직접 설계·운영하면서 프로세스 전체를 제어할 수 있게 됐습니다.
