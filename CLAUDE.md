# CLAUDE.md

## Project Overview

GitHub Pages로 배포되는 정적 리포트 사이트. HTML 기반의 인터랙티브 보고서를 호스팅한다.

- **배포**: GitHub Pages (main 브랜치)
- **구조**: `docs/` 디렉토리에 독립 HTML 파일들
- **언어**: 한국어 (Korean)

## Andrej Karpathy's Agent Guidelines

Karpathy의 에이전트 협업 원칙을 이 프로젝트에 적용한다.

### 1. 파일은 작게 유지하라

- HTML 파일 하나가 500줄을 넘기면 CSS/JS를 별도 파일로 분리할 것
- 현재 단일 파일에 스타일+스크립트+마크업이 전부 들어 있음 — 새 파일 추가 시 분리 구조 권장
- 작은 파일 = LLM이 전체 맥락을 한 번에 파악 가능

### 2. 이름을 명확하게 지어라

- 파일명: `kebab-case`, 목적이 드러나도록 (예: `ai-erd-process-report.html`)
- CSS 클래스: BEM 또는 의미 기반 네이밍
- JS 함수/변수: 역할이 즉시 파악되는 이름

### 3. 프로젝트 구조를 단순하게 유지하라

```
/
├── CLAUDE.md          # 이 파일 — 에이전트와 사람 모두를 위한 컨텍스트
├── docs/
│   ├── index.html     # 메인 대시보드 / 목차
│   ├── ai-erd-process-report.html
│   └── docflow-infographic.html
└── assets/            # (필요 시) 공유 CSS, JS, 이미지
    ├── css/
    ├── js/
    └── img/
```

### 4. 컨텍스트 파일을 최신으로 유지하라

- 이 `CLAUDE.md`가 프로젝트의 단일 진실 공급원(Single Source of Truth)
- 새 파일 추가, 구조 변경, 배포 방식 변경 시 반드시 이 파일도 업데이트
- 에이전트는 작업 시작 전 이 파일을 먼저 읽는다

### 5. 에이전트가 이해할 수 있는 코드를 작성하라

- 인라인 주석은 "왜(why)"를 설명할 때만 사용
- 매직 넘버 금지 — CSS 변수(`--navy-900` 등)와 상수 사용
- 복잡한 로직에는 단계별 주석 추가

### 6. 점진적으로 반복하라 (Iterate Incrementally)

- 한 번에 하나의 변경만 수행
- 변경 후 브라우저에서 확인 가능한 상태를 유지
- 큰 변경은 작은 커밋 단위로 분리

### 7. 테스트 가능한 상태를 유지하라

- HTML은 W3C Validator 기준 valid해야 함
- 로컬에서 `open report/index.html`로 즉시 확인 가능해야 함
- 외부 CDN 의존성 최소화 — 오프라인에서도 기본 렌더링 가능 권장

### 8. 되돌릴 수 있게 작업하라

- 파일 삭제 전 git commit
- 대규모 리팩터링은 별도 브랜치에서 진행
- `git diff`로 변경 범위를 항상 확인

## 코딩 행동 지침 (Karpathy Rules)

LLM 코딩의 흔한 실수를 줄이기 위한 행동 지침. 사소한 작업에는 판단에 따라 유연하게 적용.

### 1. 코딩 전에 생각하라

**가정하지 말 것. 혼란을 숨기지 말 것. 트레이드오프를 드러낼 것.**

구현 전:
- 가정을 명시적으로 밝힌다. 불확실하면 질문한다.
- 여러 해석이 가능하면 제시한다 — 임의로 선택하지 않는다.
- 더 단순한 접근이 있으면 말한다. 필요하면 반대 의견을 낸다.
- 불명확한 것이 있으면 멈추고, 무엇이 불명확한지 짚고, 질문한다.

### 2. 단순함 우선

**문제를 해결하는 최소한의 코드. 추측성 코드 금지.**

- 요청하지 않은 기능을 추가하지 않는다.
- 한 번만 쓰는 코드에 추상화를 만들지 않는다.
- 요청하지 않은 "유연성"이나 "설정 가능성"을 넣지 않는다.
- 불가능한 시나리오에 대한 에러 처리를 하지 않는다.
- 200줄로 쓴 것이 50줄로 가능하면 다시 쓴다.

자문: "시니어 엔지니어가 이것을 과도하게 복잡하다고 할까?" 그렇다면 단순화한다.

### 3. 외과적 변경

**건드려야 할 것만 건드린다. 자기가 만든 문제만 정리한다.**

기존 코드 수정 시:
- 인접한 코드, 주석, 포맷팅을 "개선"하지 않는다.
- 고장나지 않은 것을 리팩토링하지 않는다.
- 기존 스타일에 맞춘다 — 본인이라면 다르게 했더라도.
- 관련 없는 데드 코드를 발견하면 언급만 한다 — 삭제하지 않는다.

본인의 변경으로 고아가 된 경우:
- 본인의 변경으로 미사용된 import/변수/함수는 제거한다.
- 기존에 있던 데드 코드는 요청 없이 제거하지 않는다.

검증 기준: 변경된 모든 줄이 사용자의 요청에 직접 연결되어야 한다.

### 4. 목표 기반 실행

**성공 기준을 정의하고, 검증될 때까지 반복한다.**

작업을 검증 가능한 목표로 변환:
- "유효성 검증 추가" → "잘못된 입력 테스트 작성 후 통과시키기"
- "버그 수정" → "재현 테스트 작성 후 통과시키기"
- "X 리팩토링" → "리팩토링 전후로 테스트 통과 확인"

다단계 작업 시 간단한 계획 제시:
```
1. [단계] → 검증: [확인사항]
2. [단계] → 검증: [확인사항]
3. [단계] → 검증: [확인사항]
```

명확한 성공 기준은 독립적 반복을 가능하게 한다. 약한 기준("되게 해줘")은 지속적 확인이 필요하다.

## 팀 운영 원칙

에이전트 팀은 실제 소프트웨어 개발팀처럼 운영한다. PM이 팀 리더이며, 모든 의사결정과 조율의 중심이다.

### 보고 체계

- 에이전트는 작업 완료 또는 블로커 발생 시 **PM에게만 보고**한다.
- PM이 다음 단계를 결정한다. 에이전트가 독자적으로 다른 에이전트를 호출하지 않는다.
- 보고 내용: 완료 요약 (1~2문장) + 변경/생성 파일 목록 + 블로커 (있는 경우)

### 전파 체계

- 한 에이전트의 작업 결과가 다른 에이전트에 영향을 미치면, PM이 해당 정보를 관련 에이전트에게 전파한다.
- 전파 시 **영향받는 내용만 발췌**하여 전달한다 (전체 결과물 전달 금지).

### 도메인 위임

- 에이전트가 자신의 전문 영역 밖 작업을 발견하면, 직접 처리하지 않고 **PM에게 위임 요청**한다.
- PM이 적합한 에이전트를 선택하여 작업을 할당한다.

### 병렬 작업 관리

- 독립적인 작업은 PM이 병렬로 할당할 수 있다.
- **의존 관계가 있으면 반드시 순차 실행**한다. PM이 의존성을 판단한다.
- 병렬 작업 간 충돌 가능성이 있으면 PM이 작업 범위를 사전에 명확히 분리한다.
- 병렬 작업 완료 후 PM이 결과를 통합 검토한다.

## Code Conventions

- **HTML**: 시맨틱 태그 우선 (`<section>`, `<article>`, `<nav>`)
- **CSS**: CSS 변수(`:root`)로 디자인 토큰 관리, 인라인 `<style>` 블록 사용
- **JS**: 바닐라 JS 선호, 프레임워크 없음
- **인코딩**: UTF-8, `lang="ko"`
- **반응형**: 모바일 우선 미디어 쿼리

## Deployment

- `main` 브랜치에 push하면 GitHub Pages가 자동 배포
- `docs/` 디렉토리가 서빙 경로 (GitHub Pages 설정)
- 빌드 과정 없음 — 정적 HTML 그대로 서빙

## Do NOT

- `node_modules/`, 빌드 도구, 번들러를 도입하지 마라
- `.idea/` 등 IDE 설정 파일을 커밋하지 마라
- 기존 보고서의 데이터 수치를 에이전트가 임의로 변경하지 마라
- 외부 API 호출이 필요한 기능을 추가하지 마라 (정적 사이트 유지)
